# 第 11 章：输出区域与虚拟数据

汇编器经常要同时回答两个问题：这个值处在什么逻辑地址？它的字节会不会出现在输出文件中，如果会，又位于哪个文件偏移？

简单的 flat binary 往往让两者都从零开始并同步前进。结构化二进制则经常把它们拆开：某块内容可以加载到高地址，却仍然靠近文件开头；预留内存可以扩大地址范围而不增加文件字节；临时生成的数据也可能完全不进入最终文件。

本章说明输出区域如何表达这些情况，以及逻辑地址、文件偏移、物理文件字节和临时布局区域之间的关系。

XIRASM 用输出区域来建模这些坐标。

## 逻辑坐标与物理坐标

每个普通区域有四条属性：

- **origin**：标号地址的基准；
- **file offset**：区域字节在文件中的物理起点；
- **logical size**：包含预留空间在内的逻辑地址范围；
- **file size**：最终实际进入文件的字节数。

这四项有意保持独立。改变 origin 不会插入文件填充；改变 file offset 也不会改变标号值。

常用查询：

| 查询                        | 含义                           |
| --------------------------- | ------------------------------ |
| `region_base()`           | 当前区域的 origin              |
| `here()`                  | 当前逻辑地址                   |
| `file_offset()`           | 当前实际文件位置 |
| `file_cursor_real()`      | 当前已物理化的文件光标 |
| `file_cursor_potential()` | 如果当前预留尾部都物理化时的文件位置 |
| `tail_reserve_size()`     | 当前区域尾部尚未物理化的预留字节数 |

这些光标查询主要用于自定义布局和可复用的格式辅助函数。普通代码通常只需要标号和 `here()`。

## 设置 Origin

`origin(address)` 设定当前区域的地址起点：

```asm
// 把当前输出区域的逻辑起点设为 0x4000。
origin(0x4000);

start:
// 在起始地址处写出一个字节。
emit.u8(0xaa);

// 检查区域起点、标号地址、当前位置和文件位置。
assert(region_base() == 0x4000);
assert(label_addr("start") == 0x4000);
assert(here() == 0x4001);
assert(file_offset() == 1);
```

输出一字节：

```text
aa
```

origin 改变的是标号地址基准，不是文件中的物理位置。简单 flat 区域通常先设 origin，再定义标号或写出内容。之后再次调用 `origin` 会更新整个当前区域的基准，而不是创建新区域。

## 创建显式区域

`region.begin(name, origin, file_offset)` 同时设逻辑地址和文件偏移：

```asm
// 头部的逻辑地址从 0x1000 开始，文件位置从 0 开始。
region.begin("header", 0x1000, 0);

header:
emit.bytes(b"HD");

// 数据使用独立的逻辑地址，并从文件偏移 0x10 开始。
region.begin("payload", 0x2000, 0x10);

payload:
emit.bytes(b"DATA");

// 两个标号分别使用各自输出区域的起始地址。
assert(label_addr("header") == 0x1000);
assert(label_addr("payload") == 0x2000);
```

header 占文件偏移 0 和 1，payload 从 0x10 开始。中间未写的部分在 flat 输出里补零：

```text
48 44 00 00 00 00 00 00 00 00 00 00 00 00 00 00
44 41 54 41
```

开始另一个区域会切换当前输出区域。区域之间不是词法嵌套关系，也没有对应的 `end` 语句。

区域名描述源码层面的布局意图；它本身不会为任何可执行文件或目标文件格式创建 section 记录。那些记录属于格式接口的职责。

`region.begin` 也不是“插入到已有字节中间”的命令。它只是把后续输出切到一个由调用者明确指定逻辑 origin 和文件偏移的区域。区域之间的顺序、文件偏移是否重叠、是否留有物理空洞，都由调用者的布局模型负责。构造 PE/ELF/COFF 这类格式时，格式接口会替你维护这些关系；手写自定义格式时，必须自己把文件偏移、逻辑地址和最终回填字段分开记录。

当逻辑 origin 和物理文件偏移都已经明确时，使用 `region.begin`。如果只是按顺序构造文件，`output.section` 和 `output.org` 通常更稳妥，因为它们会从当前区域推导下一段的文件偏移。

## 实际光标与潜在光标

写已初始化字节时，两个光标同步推进：

```asm
// 写出一个实际存在于文件中的字节。
emit.u8(0xaa);

// 实际位置和潜在位置都前进到 1，尾部没有预留空间。
assert(file_cursor_real() == 1);
assert(file_cursor_potential() == 1);
assert(tail_reserve_size() == 0);
```

`reserve` 会推进潜在光标和逻辑地址，但不会立刻提交物理文件字节：

```asm
// 先写出一个字节，再预留三个尚未实际写入的字节。
emit.u8(0xaa);
reserve(3);

// 逻辑地址已经前进四字节，实际文件仍只有一个字节。
assert(here() == 4);
assert(file_cursor_real() == 1);
assert(file_cursor_potential() == 4);
assert(tail_reserve_size() == 3);
```

预留后面出现初始化数据时，预留范围变成文件中间的间隙：

```asm
// 后续字节使前面的预留范围变成文件中的实际间隙。
emit.u8(0xaa);
reserve(3);
emit.u8(0xbb);

// 中间间隙已经物理化，两个文件光标重新一致。
assert(file_cursor_real() == 5);
assert(file_cursor_potential() == 5);
assert(tail_reserve_size() == 0);
```

文件内容：

```text
aa 00 00 00 bb
```

这种延迟物理化的语义，让一次 `reserve` 既能表示文件中间的零填充间隙，也能表示不占文件空间的尾部预留。

## 裁掉预留尾部

`output.section(name, origin)` 从当前**实际**文件光标处开始新区域。

```asm
// 第一个区域包含一个实际字节和三个尾部预留字节。
emit.u8(0x41);
reserve(3);

// 新区域紧接最后一个实际字节，尾部预留空间不进入文件。
output.section("next", 0x2000);
emit.u8(0x42);
```

尾部预留会增大第一区域的逻辑大小，但新区域从最后一个已物理化字节之后开始。输出：

```text
41 42
```

适用于运行时需要地址范围但文件末尾不占空间的场景。

`output.section` 只丢弃尚未物理化的尾部预留，文件中间的间隙不受影响。

## 保留潜在间隙

`output.org(name, origin)` 从当前**潜在**光标处开新区域。

```asm
// 第一个区域的潜在长度为四字节。
emit.u8(0x41);
reserve(3);

// 新区域从潜在位置开始，因此保留前面的三字节间隔。
output.org("next", 0x2000);
emit.u8(0x42);
```

预留之后写入数据，预留范围保留在文件中：

```text
41 00 00 00 42
```

区别：

| 操作               | 下一个文件偏移起点     |
| ------------------ | ---------------------- |
| `output.section` | 实际光标，裁尾部预留   |
| `output.org`     | 潜在光标，保留预留范围 |

两个操作都会独立选择新的逻辑 origin；这个选择与文件偏移无关。

## 对齐区域文件大小

`region.file_align(alignment)` 对齐当前区域最终的物理文件大小。

```asm
// 第一个区域从逻辑地址 0x1000、文件偏移 0 开始。
region.begin("first", 0x1000, 0);

// 三个实际字节之后还有十三个尾部预留字节。
emit.bytes(b"ABC");
reserve(13);

assert(here() == 0x1010);
assert(file_cursor_real() == 3);
assert(file_cursor_potential() == 16);

// 去除尾部预留空间，再把实际文件大小向上对齐到八字节。
region.file_align(8);

// 第二个区域从已经对齐的文件偏移 8 开始。
region.begin("second", 0x2000, 8);
emit.u8(0x5a);
```

裁掉尾部预留后，三个物理字节按 8 字节边界对齐。第二区域起始文件偏移 8：

```text
41 42 43 00 00 00 00 00 5a
```

文件对齐不会推进逻辑大小。本例中，第一区域仍然描述 16 个逻辑字节。

对齐值必须是非零的 2 的幂。调用 `region.file_align` 会关闭当前区域的物理输出；继续写字节前，需要先开始另一个区域。

它不同于 `align`：`align` 会推进逻辑位置，并在间隙物理化时写出填充字节；`region.file_align` 只把最终物理大小向上取整，不改变逻辑地址。

## 虚拟输出

虚拟区域是汇编期间的隔离输出区域。它的字节、标号和指令可以被读取或转换，但不会自动复制到最终 flat 输出中。

`virtual.begin(origin)` 在指定逻辑地址创建虚拟区域：

```asm
// 在逻辑地址 0x3000 创建一个临时虚拟区域。
virtual.begin(0x3000);

table:
emit.u32(0x11223344);
// 读取原值、逐位变换后，再写回同一位置。
store.u32(table, load.u32(table) ^ 0x01010101);
const encoded: bytes = load.bytes(table, 4)

virtual.end();

// 只有显式复制出的字节才会进入主输出。
emit.bytes(encoded);
```

临时区域初始内容：

```text
44 33 22 11
```

变换后复制到主输出：

```text
45 32 23 10
```

虚拟区域本身不贡献文件字节。

虚拟区域里可以正常写出、预留、对齐、定义标签、写 ISA 指令。`load.*` 和 `store.*` 对其字节同样生效。每个 `virtual.begin` 对应一个 `virtual.end`。

虚拟区域只能作为临时输出空间使用。它不能在里面启动主输出区域：`output.section` 和 `output.org` 必须等回到主输出后再调用。虚拟区域里的文件光标也不能当成最终文件存储位置来推导；需要最终文件偏移时，应把虚拟字节显式复制到真实区域，再在稳定阶段查询真实区域的事实。

## 使用当前地址作为虚拟 Origin

省略 origin 参数时，虚拟区域从外围区域的当前逻辑地址开始：

```asm
// 主输出从逻辑地址 0x4000 开始，并先写出一个字节。
origin(0x4000);
emit.u8(0xaa);

// 省略参数，使虚拟区域沿用当前逻辑地址 0x4001。
virtual.begin();

scratch:
emit.u16(0x1234);
const copied: bytes = load.bytes(scratch, 2)

virtual.end();

// 显式把临时生成的两个字节追加到主输出。
emit.bytes(copied);
```

`scratch` 逻辑地址 0x4001，但其虚拟字节不替换也不推进外围区域。`virtual.end` 后主输出从暂停处继续。最终文件：

```text
aa 34 12
```

适用于编码需临时组装、且标签地址需与目标位置共用同一基准的场景。

## 稳定的区域事实

普通写出阶段只有实时光标可用。区域的最终文件偏移、物理大小和逻辑大小要等布局稳定后才确定。以下查询只在收尾处理阶段可用：

| 查询                             | 对包含指定地址的区域返回 |
| -------------------------------- | ------------------------ |
| `region_file_offset(address)`  | 区域在文件中的基偏移     |
| `region_file_size(address)`    | 最终物理大小             |
| `region_logical_size(address)` | 最终逻辑大小             |

这些查询和实时光标故意分开。收尾处理可以用它们回填文件头，或断言自定义格式声明的大小与最终布局一致。下一章会介绍这个阶段及其限制。

## 选择合适的工具

只用 origin：

- 单个 flat 流需非零地址基准
- 文件位置和逻辑位置同步推进

`output.section`：

- 下一段应跟在已物理化字节之后
- 尾部预留应增加内存大小但不占文件空间

`output.org`：

- 下一段应从潜在位置继续
- 后续已初始化数据之前的预留空间必须留在文件中

`region.begin`：

- 逻辑 origin 和物理文件偏移都明确
- 自定义二进制布局或格式辅助函数

虚拟区域：

- 临时组装、测量、读取或转换数据
- 临时字节不得自动进入主输出

总结：

- 标号和 `here()` 描述逻辑地址
- 实际文件光标只反映已物理化的字节
- 潜在光标包含当前尾部预留
- 最终区域事实属于稳定后的收尾处理阶段

下一章介绍收尾处理：读取已稳定的布局信息、修补字节、计算校验和、验证映像，不改变布局。
